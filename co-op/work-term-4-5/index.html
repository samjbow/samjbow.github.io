Learning Goals

Unlike previous work reports which only covered one semester, this report covers two, and thus I have two sets of learning goals to share here.

Work Term 4

  1. Goal 1: Expand my knowledge of Git

    Git, and version control software in general, are nothing new to me. In the past I'd used Git, SVN and Mercurial version control software on multiple occasions. Despite this, my comfort level with Git has always been lukewarm at best, especially when it came to dealing with edge cases where Git would do something unexpected. In the first few weeks, I'd done something to my local branches that had me pulling my hair out after merging in features to my working branch that one of my coworkers had committed. It was at this point that I enlisted the help of Christian, our automation test developer, to help me out of my plight.

    Christian taught me about the git rebase command. In short, the command takes changes in a target branch that aren't in your current branch, and applies them before your working changes. For feature branches that are only being worked on by individual developers, this is a far cleaner and more convenient method of keeping your branches up-to-date, avoiding merge conflicts, and keeping your commit tree concise. Additionally, the -i flag can be applied to perform the rebase in interactive mode, which we primarily used to squash long chains of commits made for each feature into a single commit. This again helped us keep our commit trees clean and also made identifying commits related to specific features much easier.

    In addition to learning how to manage feature branches with rebase, another useful tool I learned this year was cherry-pick. While not quite as useful as rebase, this command lets you pull individual commits into your branch. This was a vital tool for us in being able to test how branches would integrate together before actually integrating them. On occasion, we had moments where two developers would be simultaneously working on features that could only be tested in the context of one-another. As we always wanted to test before merging, we used the cherry-pick feature to pick both commits into a new testing branch and test them together before merging each branch individually.

    While not a comprehensive list of everything I did with Git, these two examples serve to show how much more confident I've become using the software. In the future I hope to expand my experience even further, but I'm extremely happy with the progress I've made with Git this work term.

  2. Goal 2: Learn to work with an Agile development team

    The biggest difference between this time at eSentire and the previous summer's time has been the change in working environment. Last year I answered almost entirely to Roy, and worked mostly either by myself or in collaboration with him. This year I was able to work with a newly formed development team, using Agile practices to manage the project we worked on. We worked in sprints, organized features and bugs into tickets, and prioritized those tickets to determine which would go into each sprint. We always thought one sprint ahead, keeping our current sprint progress in mind when considering what tickets could be added to our next sprints, and we used our experience as we grew to best gauge how much work we could handle (our velocity).

    To me, much of this hearkened back to CIS*3750 where similarly, we worked in mock sprints with features and priority for each feature. The main difference in Agile for the real world is that we are constantly modifying our sprints as needed; bugs would be discovered, tech debt would require time and clients would request specific features, all requiring sprints to be refactored to best accommodate our clients' needs. This turned out to be an integral part of sprint planning that CIS*3750 is unable to cover.

    While it may sound like I'm being harsh towards CIS*3750, the course did give me a lot of necessary vocabulary and understanding that I could bring to the table at eSentire. With a solid understanding of velocity and priority, and experience organizing our own sprints, I quickly felt comfortable in eSentire's Agile development setting.

  3. Goal 3: Become a Node.js master

    While the summer of 2015 was spent mostly on client-side JavaScript, I was challenged this semester with learning server-side JavaScript - better known as Node.js. For the most part, this task wasn't terribly complicated. Basic understanding of JavaScript was 90% of the battle; the remaining 10% was learning how Node.js (and its package manager npm) handled modules, as well as distinguishing what would and wouldn't still work moving from client- to server-side paradigms.

    The most important thing to understand about JavaScript and Node.js is that it is single-threaded and non-blocking. Your programming methods will differ from conventional languages like Python, in that any asynchronous actions must be accounted for by ensuring responses are received before proceeding with any processing. This can be accounted for using callbacks, or if you want to be a bit fancier, Promises. Callbacks can create deep levels of nesting, which are difficult to read (and even more difficult to test and debug, especially if you are using anonymous functions; a mistake I made during the first month or so of learning). In addition to using explicitly declared functions as callbacks, Promises allow you to chain callbacks together in an elegant manner that keeps your code clean and readable, even to someone with little experience in the language. Take the following code example, using callbacks: 

    function doThis ( a, b, callback ) {
  // some processing
  callback ( a, b )
}

function doThat ( a, b ) {
  //some more processing
}

doThis ( a, b, doThat );

    This works, but is a bit clunky. Promises are much cleaner, and using the this keyword, the same example would look like this: 

    function doThis ( a, b ) {
  return new Promise ( function ( resolve, reject ) ) {
    // some processing
    resolve ( a, b );
  };
}

function doThat ( a, b ) {
  return new Promise ( function ( resolve, reject ) ) {
    //some more processing
    resolve ( a, b );
  };
}

doThis ( a, b ).then ( doThat );

    There we have a piece of code that anybody could read, and it clearly communicates the asynchronous nature of the functions being called. This is just one example of the features Node.js provides to make asynchronous programming easier. I continued to use and improve with Node.js throughout my entire eight months with eSentire, and I now consider it by far my most competent language.

Work Term 5

  1. Goal 1: Get my head in the cloud

    In my first four months with eSentire, we briefly touched on cloud computing in the form of Amazon Web Services. However, my exposure to the service was largely hands-off, and this semester I was able to dive deeper into the world of cloud computing. Amazon Web Services (or AWS for short) is a service that could arguably be credited with running the internet from behind the scenes, and I was fortunate enough to be able to experience it firsthand with the project our development team worked on. The service provides a slew of features for developers and corporations to use, and while I used over half a dozen I barely scratched the surface of what AWS offers.

    First and foremost, AWS provides tools for hosting program instances using Amazon Elastic Computing Cloud, or EC2. EC2 instances are essentially replicas of the service you're hosting, which are created and destroyed on an as-needed basis to meet the volume of traffic without users being affected. This pairs nicely with Amazon CodePipeline, which provides a means of easily deploying changed code automatically. Able to detect updates in version control software repositories, Amazon CodePipeline automatically updates your service by creating updated EC2 instances and tearing down the old ones, seamlessly updating your service without interruption.

    Another popular service offered by AWS is AWS Lambda, which provides you the opportunity to run small pieces of code without having to create your own servers. One popular application for such a service is to create a microservice-based project using Lambda to host API endpoints. Coupled with AWS API Gateway and hosting datastores in EC2 (or one of Amazon's other datastore services), one can easily create a project built on lightweight APIs and microservices. Tying into this is the Serverless application framework, which allows developers to deploy code to AWS Lambda from their desktops, to any environment, without having to interrupt their existing services.

    As mentioned in my first paragraph, this is only a small example of the power Amazon Web Services can provide a development team. The above examples are merely the ones that I was focused on during this co-op term, and while I feel well versed in them now, I'm eager to learn about AWS's many other offerings in the future.

  2. Goal 2: Put forth more independent effort

    This may seem like an odd goal considering my first semester with eSentire was spent working independently, but part of joining a team includes being able to contribute to that team, and that was something I hadn't quite experienced in a co-op term yet. Despite having completed four before this, most of my work was spent on solo projects, and collaboration in a workplace setting wasn't something I had a great deal of experience with. Whether it was sorting out merge conflicts with Git, discussing important team decisions in meetings, planning future sprints or discussing implementation details, I wanted to make sure that this semester I offered my experience as often as I could (without inflating my own ego too much).

    Moreso than any work term in the past, this one was filled with meetings. Stand-ups were daily. Initially, backlog grooming happened weekly, and sprint planning and backlog grooming happened bi-weekly, but as the team grew more experienced and comfortable we were able to condense these meetings to save time. As my experience with our technology stack grew, and as I researched more and more, I felt gradually more comfortable contributing my voice to important topics of discussion. My team was also very open-minded, and our discussions were extremely insightful and enlightening.

  3. Goal 3: Learn industry testing strategy

    When you have a growing project that is constantly being updated and refined, it becomes absolutely necessary to ensure your service quality with every release. This is where testing plays an important role in an Agile workflow. Every new feature undergoes multiple forms of testing: developers and testers, perform manual functional testing, developers write unit tests for functional and regression testing, testers work with an integration test suite to ensure integration testing, etc. While any developer worth their salt should be testing their own code, this was the first semester I was able to really learn how development teams used testing frameworks to create test suites for their projects.

    Prior to my return to eSentire I had just finished CIS*4150: Software Reliability and Testing, in which we were taught various testing methods and strategies. Whereas CIS*4150 gave me a very broad look at testing strategy, my time at eSentire instead focused on depth and application. I learned how to write proper unit tests thanks entirely to a very persistent project lead (that's you, Ralph), and was able to see the results integrated with code coverage software and a fully automated regression testing suite. Among the technologies used were many Node.js unit testing packages, including Mocha, Sinon and Chai just to name a few. Mocha provided the overall framework for writing unit tests and test suites. Sinon allowed for external dependancies to be stubbed to see how a particular unit would behave under controlled circumstances. Chai validated results in a behaviour-driven manner, keeping test suites clean and readable.

    In addition to simply writing these tests suites, every once in a while a new feature would break an old one (in our development environment, fortunately) and one of us would be tasked with quickly finding and fixing the issue. More often that not the issues were minor and quickly solved, but occasionally they would require more investigation. In these situations, I also learned to leverage known test results to determine causes of these issues. In short, testing helps to facilitate debugging.
